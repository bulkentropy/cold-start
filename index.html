Sehat MG · the quality-anchored guarantee

₹10,000 for keeping the network healthy

The second Cold-start intervention, for the 99 priority CSPs who are active but not eligible for Install MG — a separate cohort from the enrolled set the rest of this command center measures. Install MG pays on an outcome the CSP only partly controls (installs); Sehat MG pays on one they fully control (network quality). Every signed-up CSP is locked to exactly one track by their current Optical Power, split at 80%: Track A — Ilaaj (OP < 80%, task = heal the plant to ≥80%) or Track B — Fit rakhna (OP ≥ 80%, task = Samasya ka Samadhan ≥80%). Payout is fully binary — the CSP's own number in the app on the last day of the cycle decides it.

The counterintuitive part: Track B's median SLA (58.3%) is lower than Track A's (60.0%) — and SLA is Track B's task. The 12 CSPs who moved from B to A at the 80% split were unusually good at service but weak on optical, so Track A inherits CSPs who already serve well and only need an optical fix, while Track B is the group running a clean plant that answers complaints slowly. Track B's ~58% has to travel 22 points to reach its gate — it is the harder half, not the easy one.
Definitions (card 11616 logic): Optical Power = in-range pings ÷ total pings over the CSP's last 15 telemetry days. SLA / Samasya ka Samadhan = complaints resolved within the 4-hour TAT ÷ total, 60-day lookback. Telemetry moves daily — every count is accurate to its pull date and must be re-run before the cohort is frozen for payout.

L1 performance by belief cohort
Matured = only bookings whose confirm-day is 4+ days old (the 3 most recent booking-days are held out), so each stage has had time to happen — accept → confirm → install. Bookings from the last 3 days are still moving through the funnel and only drag the rates down, so the headline rate excludes them. The faint total under each rate is the full-window figure including those still-maturing bookings. Booking-anchored (bucketed by booking-confirm day). The two rows below the belief cohorts are comparisons: Non-enrolled · eligible = CSPs offered MG (per the MG Tracker CSP sheet) that didn't enrol — count is the fixed universe (Flow-1 blanks + Flow-2 not opted-in/audited) · Non-eligible = rest of the network, counted by distinct receiving CSPs (no bounded universe).
Conversion ladder — slot proposed → confirmed → installed, by cohort
Belief is a CSP property (latest in-app answer; response supersedes no-response). Metrics are the same Q11528 funnel as above, normalised per CSP per day so cohort size doesn't distort the counts. Now = the post period; Before → After = these same CSPs in 1–15 Jun (matured) vs 1 Jul onward. Belief is self-selected, not assigned — a gap shows association, not proof the intervention caused it. June booking volume per CSP was far lower than now (see bookings/CSP/day), so June rates sit on thinner bases — read the small cohorts and the confirmed-% column with caution. Cohorts under 30 CSPs are dimmed.
Definitions — sacrosanct to Metabase Q11528 (B2I funnel with attributed drops). Anchor = booking confirmed (canonical booking window, test-LCO excluded); booking → connection (days=14 window) → current TAS task. A booking counts iff that task belongs to an enrolled CSP (opted-in + audit complete) = “CSP received booking”. “CSP slot proposed” = current depth ≥ 3 (slot proposed or awaiting customer slot confirmation, or further) — read as a share of bookings received, not of accepts. “Slot confirmed by customer” = depth ≥ 4 (confirmed slot or awaiting technician, or further). Speed = task creation → ALLOCATION_ACCEPTED event (timing only). By booking day: all outcomes attach to the booking's confirm day — recent days mature upward. By event day: each event counts on the day it happened, shown as absolute counts/day (accepts and confirmations there can belong to bookings from earlier days). Only complete IST calendar days are shown. Pre = 24–30 Jun daily average (dashed); Post = from 1 Jul.
Live pipeline vs Ever reached (booking-day toggle): Live pipeline uses the current TAS state (where each booking sits now) — a rung that later fell back isn't counted, so proposal/confirm rates read where the queue is today. Ever reached uses the deepest rung each booking ever hit (high-water over its candidate rows), so a slot that was proposed or confirmed and then failed/re-farmed still counts — proposal/confirm rates rise and read true progression. Same basis as the L2 Wiom view.
