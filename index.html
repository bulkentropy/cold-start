L1 · Leading metrics — enrolled CSPs
L1 performance by belief cohort
Matured = only bookings whose confirm-day is 4+ days old (the 3 most recent booking-days are held out), so each stage has had time to happen — accept → confirm → install. Bookings from the last 3 days are still moving through the funnel and only drag the rates down, so the headline rate excludes them. The faint total under each rate is the full-window figure including those still-maturing bookings. Booking-anchored (bucketed by booking-confirm day). The two rows below the belief cohorts are comparisons: Non-enrolled · eligible = CSPs offered MG (per the MG Tracker CSP sheet) that didn't enrol — count is the fixed universe (Flow-1 blanks + Flow-2 not opted-in/audited) · Non-eligible = rest of the network, counted by distinct receiving CSPs (no bounded universe).
Conversion ladder — accepted → confirmed → installed, by cohort
Belief is a CSP property (latest in-app answer; response supersedes no-response). Metrics are the same Q11528 funnel as above, normalised per CSP per day so cohort size doesn't distort the counts. Now = the post period; Before → After = these same CSPs in 1–15 Jun (matured) vs 1 Jul onward. Belief is self-selected, not assigned — a gap shows association, not proof the intervention caused it. June booking volume per CSP was far lower than now (see bookings/CSP/day), so June rates sit on thinner bases — read the small cohorts and the confirmed-% column with caution. Cohorts under 30 CSPs are dimmed.
Definitions — sacrosanct to Metabase Q11528 (B2I funnel with attributed drops). Anchor = booking confirmed (canonical booking window, test-LCO excluded); booking → connection (days=14 window) → current TAS task. A booking counts iff that task belongs to an enrolled CSP (opted-in + audit complete) = “CSP received booking”. “CSP accepted / slot proposed” = current depth ≥ 3 (slot proposed or awaiting customer slot confirmation, or further). “Slot confirmed by customer” = depth ≥ 4 (confirmed slot or awaiting technician, or further). Speed = task creation → ALLOCATION_ACCEPTED event (timing only). By booking day: all outcomes attach to the booking's confirm day — recent days mature upward. By event day: each event counts on the day it happened, shown as absolute counts/day (accepts and confirmations there can belong to bookings from earlier days). Only complete IST calendar days are shown. Pre = 24–30 Jun daily average (dashed); Post = from 1 Jul.
Live pipeline vs Ever reached (booking-day toggle): Live pipeline uses the current TAS state (where each booking sits now) — a rung that later fell back isn't counted, so proposal/confirm rates read where the queue is today. Ever reached uses the deepest rung each booking ever hit (high-water over its candidate rows), so a slot that was proposed or confirmed and then failed/re-farmed still counts — proposal/confirm rates rise and read true progression. Same basis as the L2 Wiom view.
